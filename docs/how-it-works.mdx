# How It Works

When you run `npm run test:headed` inside your `.openqa/` directory, this is what happens:

<Steps>
  <Step title="Your BDD step fires">
    Playwright-BDD or Cucumber.js triggers a step from your `.feature` file, e.g.:
    ```
    * I add a new todo item "Buy groceries"
    ```
  </Step>
  <Step title="runAgent() is called">
    The generated step definition in `.openqa/steps/steps.ts` passes the step text to `runAgent()`.
  </Step>
  <Step title="The harness spawns Claude Code">
    OpenQA spawns a Claude Code CLI subprocess (`npx @anthropic-ai/claude-code`) with a per-run `.mcp.json` that points Claude at a live TCP bridge to your Playwright browser.
  </Step>
  <Step title="Claude drives the browser">
    Claude receives the natural language instruction and uses Playwright MCP tools (`browser_navigate`, `browser_click`, `browser_type`, etc.) to drive the real browser.
  </Step>
  <Step title="The step passes or fails">
    If Claude reports success, the step passes. If a tool call errors or Claude makes zero tool calls, the step fails immediately.
  </Step>
</Steps>

## Key Design Decisions

**Browser sharing** — Claude drives the *exact same* Playwright page object that your test holds. There's no separate browser window. This means shared cookies, session storage, and page state.

**Parallel-safe** — Each test worker gets its own unique TCP port and `.mcp.json` file in a temporary directory. Tests run in parallel with zero config collisions.

**Session continuity** — Within a single test scenario, Claude resumes its conversation session across steps. It remembers the page it navigated to, so each subsequent step doesn't need to re-explain context.

**No SDK in your test process** — The heavy AI model runs as a subprocess. Your test process only holds a Playwright browser and a lightweight orchestrator.

## Authentication

OpenQA checks for credentials in this order:

| Method | How |
|--------|-----|
| `claude login` session | Best for local development — no key needed |
| `ANTHROPIC_API_KEY` | Set in `.openqa/.env` or shell environment |
| `CLAUDE_CODE_OAUTH_TOKEN` | For CI pipelines with OAuth |

The `.openqa/.env.example` file (generated by `openqa init`) has the template.
